Diferentes mundos de captura de datos y
Análisis de los datos
Uno de los activos más importantes de cualquier organización es su información. Este activo
casi siempre se usa con dos propósitos: mantenimiento de registros operativos y análisis
Toma de decisiones. 
Estos sistemas están optimizados para consultas de alto rendimiento como preguntas de los usuarios a menudo
requieren que se busquen cientos o cientos de miles de transacciones y
comprimido en un conjunto de respuestas. Para complicar aún más las cosas, los usuarios de un DW / BI
sistema típicamente demanda que se preserve el contexto histórico para evaluar con precisión
el rendimiento de la organización a lo largo del tiempo.

Estos temas recurrentes han existido por más de tres
décadas:
¦ "Recopilamos toneladas de datos, pero no podemos acceder a ellos".
¦ "Necesitamos dividir y cortar los datos de todas formas".
¦ "Las personas de negocios necesitan obtener los datos fácilmente".
¦ "Solo enséñame lo que es importante".
¦ "Pasamos reuniones enteras discutiendo sobre quién tiene los números correctos en lugar de
que tomar decisiones ".
¦ "Queremos que las personas usen la información para apoyar una decisión más basada en hechos
fabricación."

El sistema DW / BI debe ser diseñado para manejar este cambio inevitable correctamente para que no invalide
datos o aplicaciones existentes. Los datos y aplicaciones existentes no deberían
cambiarse o interrumpirse cuando la comunidad empresarial haga nuevas preguntas
o se agregan nuevos datos al almacén. 

***Star Schemas Versus OLAP Cubes
Los modelos dimensionales implementados en los sistemas de gestión de bases de datos relacionales son
conocidos como esquemas de estrellas debido a su parecido con una estructura de estrella.
Los modelos dimensionales implementados en entornos de bases de datos multidimensionales son
conocidos como cubos de procesamiento analítico en línea (OLAP)

***
Consideraciones de implementación de OLAP
-Un esquema en estrella alojado en una base de datos relacional es una buena base física
para construir un cubo OLAP, y generalmente se considera como una base más estable
para copia de seguridad y recuperación.
-Las estructuras de datos de cubos OLAP son más variables en diferentes proveedores que
DBMS relacionales, por lo que los detalles finales de implementación a menudo dependen de
Se elige el proveedor OLAP. 
-Los cubos OLAP suelen tener opciones de seguridad más sofisticadas que los RDBMS,
como limitar el acceso a datos detallados pero proporcionando un acceso más abierto a
datos resumidos.
-Los cubos OLAP ofrecen capacidades de análisis significativamente más ricas que los RDBMS
-Los cubos OLAP pueden imponer restricciones detalladas en la estructura de la dimensión
claves que implementan jerarquías detalladas en comparación con las bases de datos relacionales.
-Algunos productos OLAP no habilitan roles o alias dimensionales, por lo que requieren
dimensiones físicas separadas para ser definidas.

***
Tablas de datos para mediciones
La idea de que un evento de medición en el mundo físico tiene un uno-a-uno
relación con una sola fila en la tabla de hechos correspondiente es un principio fundamental
para el modelado dimensional. Todo lo demás se construye desde esta base.
Los hechos más útiles son numéricos y aditivos, como el monto de las ventas en dólares.

Debieras
no almacenar información textual redundante en tablas de hechos. A menos que el texto sea único
para cada fila en la tabla de hechos, pertenece a la tabla de dimensiones. 

***Tablas de dimensiones para el contexto descriptivo
Las tablas de dimensiones son componentes integrales de una tabla de hechos. Las tablas de dimensiones contienen
el contexto textual asociado con un evento de medición del proceso de negocio. Ellos
describir el "quién, qué, dónde, cuándo, cómo y por qué" asociado con el evento.
Los atributos de dimensión sirven como fuente primaria de restricciones de consulta, agrupaciones,
e informe las etiquetas. En una consulta o solicitud de informe, los atributos se identifican como
por palabras. Por ejemplo, cuando un usuario desea ver las ventas en dólares por marca, la marca debe
estar disponible como un atributo de dimensión

La Figura 1-4 muestra que las tablas de dimensiones a menudo representan relaciones jerárquicas.
Por ejemplo, los productos se acumulan en marcas y luego en categorías. Para cada
fila en la dimensión del producto, debe almacenar la marca y categoría asociadas
descripción. La información descriptiva jerárquica se almacena de forma redundante en
espíritu de facilidad de uso y rendimiento de consultas. Deberías resistirte al quizás habitual
instar a normalizar los datos almacenando solo el código de marca en la dimensión del producto y
creando una tabla de búsqueda de marca por separado, y también para la descripción de la categoría en una
tabla de búsqueda de categoría separada. Esta normalización se llama snowflking.

***Extract, Transformation, and Load System
La extracción es el primer paso en el proceso de obtener datos en el almacén de datos
ambiente. Extraer significa leer y comprender los datos fuente y
copiando los datos necesarios en el sistema ETL para una mayor manipulación. En esto
punto, los datos pertenecen al almacén de datos.
Después de que los datos se extraen al sistema ETL, hay numerosas transformaciones potenciales,
como limpiar los datos (corregir errores ortográficos, resolver el dominio

datos de múltiples fuentes y datos de deduplicación.

NOTA: es aceptable crear una base de datos normalizada para admitir ETL
procesos; sin embargo, este no es el objetivo final. Las estructuras normalizadas deben ser
fuera de los límites de las consultas de los usuarios porque derrotan los objetivos gemelos de la comprensibilidad
y rendimiento
